* До появления обобщений все решалось через Object, но это было небезопасно.
* Чтобы не получить исключения при реализации небезопасного класса, надо делать проверку на instanceof (и таких проверок может быть очень много) - это неудобно.
* Поэтому были придуманы Generic (Обобщения).

* С использованием Обобщения мы делаем наш класс безопасным.
* Если мы строго типизируем класс, например < String >, то при попытке использования объектов другого типа получим ошибку компиляции .

* Пример класса Repository, который используется во фреймворке Spring для работы с базой данных.

* Пример класса MyMath с типизированным статическим методом getMiddle, при этом сам класс может быть необобщенным.
* Важно! В качестве дженерика может выступать только ссылочный тип.
* Внутри класса с типизированным методом можно использовать и обычные методы (say()).
* Если внутри класса или интерфейса мы хотим использовать типизированное поле (E item) нужно типизировать и сам класс.

* Чтобы применить метод сравнения в обобщенном классе (compareTo), просто дженерика будет недостаточно, нужно его расширять интерфейсом Comparable.
* Вообще любой обобщенный тип можно расширить одним классом (как в наследовании) и любым количеством интерфейсов. В примере выше класс People и интерфейсы Comparable, Serializable.

* Чтобы использовать объект для сравнения в дженерик классе, ограниченном интерфейсом, нужно соответственно имплементировать в класс объекта этот интерфейс (People implements Comparable< People >) и переопределить метод compareTo.

* Обобщенный тип при компиляции преобразуется в байт-коде в Object.
* Если в дженерике есть ограничения (extends), в байт-коде при копиляции он преобразуется в тип ограничения (на рис. выше в тип Comparable).

* Пример правильной инициализации массива обобщенного типа.

* Если класс типизирован, нельзя использовать переменную обобщенного типа в статическом контексте.
* Это объясняется тем, что когда мы создаем дженерик, мы его по сути инициализируем (без инициализации его использовать нельзя), а статический контекст может проявиться раньше, чем инициализированный дженерик.

* Нельзя перехватить исключения обобщенного типа, потому что в рантайме отличия объектов разного типа стираются (будь то String, Integer или что-то еще), все объекты будут типа Object, т.е. перехват исключения теряет смысл.

* По причине того, что в рантайме отличия объектов разного типа стираются будут отсутствовать отношения наследования между типами, подставляемыми в дженерик, классы которых являются наследниками.
* И Pair< Employee > и Pair< Manager > будут одним классом Pair.
